# Тестирование и апробация

Руководство по запуску всех проверок проекта и по целевым метрикам апробации (этап 14 плана реализации, раздел 3.6 ВКР). Архитектура системы — в [architecture.md](architecture.md).

## 1. Слои тестирования

| Слой | Где | Что проверяет |
|------|-----|---------------|
| Модульные тесты микросервиса | `Микросервис_IRT/tests/test_*.py` | модели IRT: вероятность и информация Фишера, оценка θ (MLE/MAP), селектор, калибровка (Rasch/JMLE/MMLE) |
| Интеграционные тесты микросервиса | `Микросервис_IRT/tests/integration/test_endpoints.py` | сквозные HTTP-эндпоинты, контракт ключей запроса/ответа, сходимость на синтетических данных, калибровка через API |
| Сценарии конфигурации (xUnitFor1C) | `Конфигурация/tests_1c/` | связка общих модулей 1С: сессия теста, бейджи, траектория, рейтинг, калибровка |
| Генератор синтетических данных | `Конфигурация/fixtures/synthetic_data_generator.py` | формирование данных с известным θ_true для тестов и апробации |

## 2. Полный прогон автоматических проверок

Скрипт `deploy/run_all_tests.sh` (или `run_all_tests.ps1` для Windows) последовательно выполняет линтер, тесты, типизацию и дымовой запуск генератора.

```bash
# Linux / macOS / Git Bash
cd Конфигурация/deploy
bash run_all_tests.sh
```

```powershell
# Windows PowerShell
cd Конфигурация\deploy
powershell -ExecutionPolicy Bypass -File run_all_tests.ps1
```

Шаги: `ruff check` (блокирующий) → `pytest` модульные + интеграционные (блокирующий) → `mypy app` (информационный) → дымовой запуск генератора (блокирующий). Сценарии 1С запускаются отдельно (см. §5).

## 3. Тесты микросервиса напрямую

Интерпретатор выбирается автоматически; на машинах без системного Python используйте `uv` (см. примечание в конце):

```bash
cd Микросервис_IRT
uv pip install -e ".[dev]"   # или: pip install -e ".[dev]"

uv run pytest                # все тесты (модульные + интеграционные)
uv run pytest tests/integration   # только интеграционные
uv run ruff check app tests  # линтер
uv run mypy app              # типизация (см. §6)
```

Интеграционные тесты используют `TestClient` FastAPI (на базе httpx): приложение поднимается через ASGI без реального сетевого порта, поэтому отдельный запуск `uvicorn` для них не нужен.

## 4. Генерация синтетических данных

Генератор строит банк (по умолчанию читает реальные `fixtures/question_bank/*.json`), модельных испытуемых с известным θ_true, полную матрицу для калибровки и истории адаптивных сессий.

```bash
cd Конфигурация/fixtures
uv run python synthetic_data_generator.py --n-users 100 --seed 20260531 --out synthetic
```

Параметры: `--n-users` (число испытуемых), `--seed` (детерминизм), `--test-length` (длина сессии), `--grid` (θ_true по равномерной сетке вместо N(0,1)), `--out` (каталог вывода).

Выходные файлы:

- `synthetic_bank.json` — банк с истинными b и a;
- `synthetic_respondents.json` — θ_true каждого испытуемого;
- `calibration_matrix.json` — тело запроса для `/calibrate_bank` (ключи `Тема`, `Метод`, `Матрица`);
- `synthetic_sessions.json` — истории ответов адаптивных сессий;
- `dataset_meta.json` — параметры генерации.

Файлы пригодны как для интеграционных тестов микросервиса, так и для загрузки в тестовую информационную базу 1С при апробации. Модуль также импортируется интеграционными тестами как библиотека (функции `make_synthetic_bank`, `make_respondents`, `generate_calibration_matrix`, `simulate_response`).

## 5. Сценарии конфигурации (xUnitFor1C)

Пять интеграционных сценариев 1С (базовая сессия, бейдж, траектория, рейтинг, калибровка) описаны в [`../tests_1c/`](../tests_1c/README.md). Они требуют информационной базы с загруженной конфигурацией и фреймворка xUnitFor1C; в автоматический прогон `run_all_tests` не входят и запускаются вручную из «Тестера».

## 6. Целевые метрики апробации

Соответствуют таблице 3.5 и метрике М-7 раздела 3.6 ВКР. Проверяются интеграционными и модульными тестами на синтетических данных с известным θ_true:

| Метрика | Целевое значение | Где проверяется |
|---------|------------------|-----------------|
| М-7: RMSE оценки θ при длине теста 15 | ≤ 0,30 логит | `test_selector.py::test_adaptive_session_converges`, `test_endpoints.py::test_full_session_rmse_at_length_15` |
| Средняя длина адаптивной сессии | ≤ 18 заданий | `test_endpoints.py::test_full_session_stops_within_limit` |
| Сходимость (потолок длины сессии) | ≤ 20 заданий, типично ≤ 15 | там же |
| Восстановление параметров банка при калибровке | corr(b̂, b) > 0,90 на 200 испытуемых | `test_calibration.py`, `test_endpoints.py::test_calibrate_bank_endpoint_recovers_difficulty` |

Тесты детерминированы (фиксированные seed через `numpy.random.default_rng`) и воспроизводимы независимо от `PYTHONHASHSEED`.

## 7. Известные замечания

- **mypy.** Шаг `mypy app` в скрипте — информационный (не блокирует прогон). На текущем коде он сообщает о замечаниях типизации в `app/irt_rasch.py` и `app/calibration.py` (сужение `float | None`, `no-any-return` от numpy). На корректность расчётов и прохождение тестов это не влияет; устранение типизации вынесено в отдельную задачу.
- **mypy и генератор.** Генератор лежит в каталоге с кириллицей (`Конфигурация/fixtures`); mypy 2.1.0 на таком пути падает с internal error, поэтому в scope типизации не входит и проверяется только линтером ruff.
- **Предупреждение httpx/starlette.** `StarletteDeprecationWarning` про httpx — некритичное предупреждение версии, на результаты не влияет.

## Примечание: Python через `uv`

На машинах без системного Python запускайте всё через `uv` (виртуальное окружение создаётся автоматически):

```bash
cd Микросервис_IRT
uv venv
uv pip install -e ".[dev]"
uv run pytest
```
