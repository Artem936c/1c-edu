# Архитектура приложения автоматизированного обучения и тестирования по платформе «1С: Предприятие» 8.3

Документ описывает общую архитектуру системы, состав подсистем, протокол взаимодействия конфигурации с микросервисом IRT, модель данных и принятые принципы кодирования. Соответствует разделам 2.3, 3.1–3.4 ВКР.

## 1. Назначение и состав системы

Система реализует адаптивное тестирование на основе моделей теории ответов на задания (Item Response Theory, IRT), построение индивидуальных траекторий обучения, геймификацию учебной деятельности и аналитические дашборды для обучающихся и преподавателей.

Система состоит из трёх согласованно работающих частей:

1. **Прикладная конфигурация «1С: Предприятие» 8.3** (`Конфигурация/`) — ядро системы: хранение данных, бизнес-логика, пользовательский интерфейс, права доступа, регламентные задания.
2. **Внешний микросервис IRT на Python/FastAPI** (`../Микросервис_IRT/`) — stateless-сервис математических расчётов: оценка способности θ, выбор следующего задания, калибровка банка.
3. **HTML-дашборды с Chart.js** (`dashboards/`) — интерактивные визуализации, встраиваемые в управляемые формы 1С через поле «Поле HTML-документа».

Разделение математического ядра в отдельный сервис обусловлено тем, что встроенный язык 1С плохо приспособлен для численной оптимизации (метод Брента, L-BFGS-B), а вынос расчётов на Python с `scipy`/`numpy` обеспечивает точность и производительность. Сервис не хранит состояние: вся история и параметры передаются в теле запроса, поэтому его можно масштабировать горизонтально и перезапускать без потери данных.

## 3. Подсистемы конфигурации

| Подсистема | Состав и назначение |
|------------|---------------------|
| **Курсы** | Справочники `Курсы`, `Темы`; навигация по учебным программам и их структуре. |
| **Тестирование** | Документ `ПопыткаТеста`, справочники `ТестовыеЗадания`, `ПрактическиеЗадания`, регистр `ПараметрыIRT`; адаптивное тестирование и проверка практических заданий. |
| **Библиотека** | Справочник `МатериалыБиблиотеки`, документ `ИзучениеМатериала`, регистр `ПривязкаМатериаловКТемам`; учебные материалы и их связь с темами. |
| **Геймификация** | Справочники `Бейджи`, `ПравилаНачисленияБаллов`, документы `НачислениеБаллов`, `ВыдачаБейджа`, регистры `БаллыОпыта`, `РейтинговыеПозиции`. |
| **Аналитика** | Регистры `СобытияАналитики`, `СостояниеТраекторииОбучения`, `ОценкиПодготовленностиПоТемам`, отчёты-дашборды. |
| **Администрирование** | Константы настройки, справочник `УчебныеГруппы`, регламентные задания, обработка `ПервоначальноеНаполнение`. |

## 4. Модель данных

### 4.1. Справочники (классификаторы предметной области)

- `Курсы` — учебные курсы (преподаватель, период, целевая аудитория).
- `Темы` — иерархический, подчинён `Курсы`; содержит сложность, порядок изучения и табличную часть пререквизитов.
- `МатериалыБиблиотеки` — учебные материалы с библиографическим описанием и привязками к темам.
- `ТестовыеЗадания` — задания с типом, формулировкой, вариантами ответа и версией параметров IRT.
- `ПрактическиеЗадания` — задания с присоединённым эталонным артефактом.
- `Бейджи` — награды с категорией и табличной частью условий активации.
- `ПравилаНачисленияБаллов` — правила «событие → баллы».
- `УчебныеГруппы` — группы обучающихся с составом и преподавателем.

### 4.2. Документы (учебные события)

- `ПопыткаТеста` — сессия адаптивного тестирования с табличной частью пошаговых ответов (θ и SE на каждом шаге).
- `ВыполнениеПрактическогоЗадания` — сдача практического задания и его проверка.
- `ИзучениеМатериала` — факт изучения материала библиотеки.
- `НачислениеБаллов` — начисление баллов опыта.
- `ВыдачаБейджа` — выдача бейджа обучающемуся.

### 4.3. Регистры

**Сведений:**

- `ОценкиПодготовленностиПоТемам` (периодический) — измерения Пользователь, Тема; ресурсы Тета, SE; источник — ссылка на `ПопыткаТеста`.
- `ПараметрыIRT` — измерения Задание, ВерсияПараметров; ресурсы ТрудностьB, ДискриминативностьA, СтатусКалибровки.
- `РейтинговыеПозиции` (периодический) — измерения УчебнаяГруппа, Пользователь; ресурсы СуммаБаллов, Ранг, СредняяТета.
- `СостояниеТраекторииОбучения` — измерения Пользователь, Курс; план обучения в формате JSON, текущая тема, процент завершения.
- `ПривязкаМатериаловКТемам` — измерения Материал, Тема; приоритет и позиция.
- `СобытияАналитики` — измерения Пользователь, Тема, ТипСобытия; время и полезная нагрузка (JSON).

**Накопления:**

- `БаллыОпыта` (оборотный) — измерения Пользователь, Тема, Причина; ресурс Количество.

## 6. Протокол взаимодействия с микросервисом IRT

### 6.1. Транспорт

Обмен — по HTTP, тело запроса и ответа — JSON в кодировке UTF-8 без BOM. Клиент реализован в `КлиентIRTСервиса.ВыполнитьЗапрос`: формирует `HTTPСоединение` (с поддержкой HTTPS через `ЗащищённоеСоединениеOpenSSL`), сериализует структуру 1С через `ЗаписатьJSON`, разбирает ответ через `ЧтениеJSON`. Адрес сервиса берётся из константы `URLМикросервисаIRT`, таймаут — из `ТаймаутЗапросаКМикросервису` (по умолчанию 10 секунд).

### 6.2. Соглашение об именах ключей

Поскольку конфигурация сериализует структуры как есть, **ключи запросов — кириллические** (имена свойств структур 1С: `ИсторияОтветов`, `Кандидаты`, `ТекущаяТета`). **Ключи ответов — латинские** (`theta`, `se`, `selected_id`, `new_params`), конфигурация читает их по этим именам. Сопоставление зафиксировано в `app/schemas.py` через `alias` Pydantic-полей.

### 6.3. Эндпоинты

| Эндпоинт | Запрос (кириллические ключи) | Ответ (латинские ключи) |
|----------|------------------------------|--------------------------|
| `POST /estimate_theta` | `ИсторияОтветов[]`, `Априор{Мю,Сигма}` | `theta`, `se`, `stop`, `n_items` |
| `POST /select_next_item` | `ТекущаяТета`, `Кандидаты[]`, `ИспользованныеЗадания[]` | `selected_id`, `info` |
| `POST /calibrate_bank` | `Тема`, `Матрица[]`, `Метод` | `new_params[]`, `method`, `n_items`, `n_users` |
| `GET\|POST /health` | — | `status` |

### 6.4. Отказоустойчивость

При недоступности микросервиса, таймауте или нештатном HTTP-коде `КлиентIRTСервиса` фиксирует сбой в журнале регистрации и возвращает признак неуспеха. Модуль `АдаптивноеТестирование` в этом случае автоматически переключается на локальный расчёт θ и выбор задания (`АдаптивноеТестированиеЛокально`) по упрощённой модели Раша — тестирование продолжается без сервиса. Регламентное задание `СинхронизацияСМикросервисомIRT` периодически опрашивает `/health` и фиксирует доступность.

## 7. Сценарий адаптивного тестирования (поток данных)

1. `НачатьСессию` создаёт документ `ПопыткаТеста`, читает начальную θ из среза последних значений регистра `ОценкиПодготовленностиПоТемам`, выбирает первое задание из банка темы.
2. На каждый ответ `ОбработатьОтвет`:
   - собирает историю ответов сессии с параметрами заданий из `ПараметрыIRT`;
   - запрашивает у микросервиса новую оценку θ и SE (`/estimate_theta`), при недоступности — локальный расчёт;
   - дописывает строку в табличную часть «Ответы» (θ и SE текущего шага);
   - регистрирует событие в `СобытияАналитики`;
   - проверяет критерии остановки: достигнута целевая SE (`ЦелеваяСтандартнаяОшибкаТета`) или лимит заданий (`ЛимитЗаданийВСессии`);
   - если не пора останавливаться — выбирает следующее задание (`/select_next_item`) по максимуму информации Фишера.
3. `ЗавершитьСессию` фиксирует итоговые θ и SE, записывает их в регистр `ОценкиПодготовленностиПоТемам`, начисляет баллы за верные ответы (`Геймификация.НачислитьБаллы`), проверяет условия бейджей и регистрирует событие завершения.

## 8. HTML-дашборды

Дашборды (`dashboards/student_dashboard.html`, `teacher_dashboard.html`, `test_report.html`) хранятся также как макеты «HTML-документ» соответствующих отчётов конфигурации. Управляемая форма передаёт данные в страницу вызовом JavaScript-функции `initDashboard(jsonData)` через метод `ВыполнитьJavaScript()` поля «Поле HTML-документа». JSON формируют функции модуля `ДанныеДашборда`. Общие обёртки над Chart.js вынесены в `dashboards/shared/chart_helpers.js`, стили — в `dashboards/shared/styles.css`.

## 9. Регламентные задания

| Задание | Расписание | Действие |
|---------|-----------|----------|
| `ОбновлениеРейтинга` | ежедневно в нерабочие часы | пересчёт `РейтинговыеПозиции` по `БаллыОпыта` |
| `КалибровкаБанкаЗаданий` | ежедневно в нерабочие часы | вызов `/calibrate_bank` для тем с накопленной историей, запись новой версии параметров |
| `ПроверкаУсловийБейджей` | каждый час | проверка накопительных условий по активным обучающимся |
| `ОчисткаУстаревшихСобытийАналитики` | еженедельно | архивирование событий старше 90 дней |
| `СинхронизацияСМикросервисомIRT` | каждые 5 минут | проба `/health`, переключение режима тестирования |

## 10. Роли и права доступа

Три роли с RLS-ограничениями (Record Level Security):

- **Обучающийся** — чтение активных курсов, тем своих курсов, привязанных материалов; чтение только собственных документов и записей регистров; запуск адаптивных сессий.
- **Преподаватель** — права обучающегося плюс изменение справочников курсов/тем/заданий/материалов/бейджей; чтение документов и регистров обучающихся своих учебных групп; запуск калибровки и обновления рейтинга.
- **АдминистраторУчебногоПроцесса** — все права преподавателя плюс управление учебными группами и пользователями, перерасчёт всех агрегатов.

## 11. Принципы кодирования

- Имена объектов метаданных и идентификаторы встроенного языка — на русском, CamelCase, согласно «1С: Стандартам разработки».
- Имена файлов в файловой системе — на латинице, разделитель `_`.
- Все исходные файлы — UTF-8 без BOM.
- Выгрузка конфигурации хранится в формате XML для корректной работы систем контроля версий.
- Микросервис — типизированный Python (Pydantic-схемы, аннотации типов), проверяется `ruff` и `mypy`, покрыт `pytest`.
- Все обращения к микросервису обёрнуты в обработку исключений с фиксацией в журнале регистрации и резервным локальным расчётом.
